## Object-store naming: pywaggle builds it, not the plugin

An object like `1783105211072377356-sample.jpg` is generated by **pywaggle**, not
the plugin. The plugin only supplies the base filename (e.g. `"sample.jpg"`).

- `Plugin.upload_file(path, meta={}, timestamp=None, keep=False)`:
  - `timestamp = timestamp or get_timestamp()`  ÔåÉ the number
  - `meta["filename"] = Path(path).name`  (pywaggle auto-injects ONLY this key)
  - object/staged name = `f"{timestamp}-{src.name}"`
  - publishes a message with reserved `name="upload"`, `value=<staged name>`,
    `timestamp`, `meta`.
- The number is a **nanosecond epoch timestamp** (`get_timestamp()` is literally
  `time.time_ns()`; see data/timestamp.py). Divide by 1e9 for seconds.
- Uniqueness comes from the nanosecond prefix, not from anything the plugin does.
- Two DISTINCT naming schemes exist ÔÇö don't confuse them:
  - object-store / staged object name = `{timestamp}-{filename}`
    ÔåÆ `1783105211072377356-sample.jpg` (what you see in the store)
  - on-node staging DIRECTORY (Uploader.upload_file in uploader.py) =
    `{timestamp}-{sha1sum}/` containing files `data` + `meta` (checksum-based,
    NOT the object name).

## What travels in meta: plugin/pywaggle vs SERVER-side enrichment

At the wire, pywaggle attaches very little; the cloud enriches the rest.

- pywaggle-published meta = plugin's dict + auto `filename`. e.g. imagesampler:
  `{"camera": "<stream>", "filename": "sample.jpg"}`. Nothing else.
- The ONLY node/job context added on the wire is the AMQP property `app_id`
  (env `WAGGLE_APP_ID`), set as a message PROPERTY (not meta). Code comment:
  "app_id is used by data service to validate and tag additional metadata
  provided by k3s scheduler."
- Therefore `vsn`/node, `host`, `job`, `task`, `plugin` (the `registryÔÇª:version`
  tag), zone, lat/lon are injected **server-side** (validator/data-service),
  keyed off `app_id` + the k3s scheduler, AFTER the message leaves the node.

## Hard constraints pywaggle enforces (bite you if ignored)

- `valid_meta()`: meta MUST be a FLAT dict of **str ÔåÆ str**. Values must be
  strings. Any numeric field (e.g. a second timestamp) must be stringified:
  `meta["capture_timestamp"] = str(sample.timestamp)`.
- `MIN_TIMESTAMP_NS` guard: `__publish` rejects any timestamp < 2000-01-01 (ns).
  This is the seconds-vs-nanoseconds unit tripwire ÔÇö hand-built timestamps must
  be nanoseconds since epoch.
- `name="upload"` is reserved: `raise_for_invalid_publish_name` blocks a plugin
  from `publish()`-ing that name itself; only `upload_file()` uses it internally.

## Which timestamp does a record get?

There is exactly ONE timestamp slot per message. It is shared by the object-name
prefix, the message `timestamp`, and the sidecar meta.

- Default (`timestamp=None`) ÔåÆ `get_timestamp()` evaluated INSIDE `upload_file()`
  = **upload time**. Wrong event: "when the file left the node," not "when the
  scene was captured." Drifts under upload lag (e.g. cross-country object-store
  propagation, batch-and-hold).
- Pass `timestamp=sample.timestamp` ÔåÆ record/name reflect the frame's grab time.
  **This is the one-line fix** to make names/records reflect capture instead of
  upload. Most plugins forget to pass it.

## RTSP sample.timestamp semantics (VERIFIED)

`sample.timestamp` is **host wall-clock time of frame GRAB (node clock), NOT
camera exposure time.** For RTSP it can be up to ~10 ms stale vs the returned
frame. It is NEVER the RTP/sensor/camera timestamp.

- `Camera.__enter__` sets `enable_daemon=True` for anything that is NOT a
  `file://` URL ÔåÆ RTSP/HTTP go through a background daemon.
- Daemon `_run()`: `grab()` then `self.timestamp = get_timestamp()` every ~10 ms.
- `snapshot()` returns the most recent daemon frame with the daemon's stored
  timestamp ÔåÆ up to ~10 ms stale.
- Additional UNMEASURED cameraÔåÆnode pipeline delay (encode + network + decoder
  buffer, tensÔÇôhundreds of ms) is not captured at all. True exposure-time
  alignment would require the RTP/ONVIF timestamp, which pywaggle does not surface.
- Other sources: USB/non-daemon fallback = host time right after a synchronous
  grab (no 10 ms staleness); `file://` video = synthesized `base + idx/fps`;
  `ImageFolder` (test-dir mode) = file `st_mtime_ns` (file mtime, not capture).

## Two-timestamp pattern for batch-and-hold

When capture and upload are decoupled (batch pull, hold-for-analysis, upload lag),
the captureÔåÆupload gap grows from ms to minutes and the single default timestamp
loses the capture instant. Carry both, in the plugin:

1. Record timestamp = capture/grab time: `upload_file(path, meta=meta,
   timestamp=sample.timestamp)`. Label it honestly as "node grab time," not
   "exposure time."
2. Preserve send time in meta: `meta["upload_timestamp"] = str(get_timestamp())`
   captured at the actual upload call (pywaggle won't record it for you).
3. Optional redundancy: `meta["capture_timestamp"] = str(sample.timestamp)`.
4. Latency metric computable at query time: `upload_timestamp - timestamp` =
   hold+stage+upload latency (directly measures the drift).

Rules: all meta values STRINGS; all timestamps NANOSECONDS ÔëÑ 2000-01-01.

### Batch-and-hold performance note

The RTSP daemon thread lives only inside `with self.capture:`, and
`Camera.snapshot()` wraps EACH call in that context ÔÇö so every snapshot spins up
the daemon, grabs, and tears it down. At high pull rates this churns the RTSP
connection. A batch puller should hold ONE `Camera`/`_Capture` open across the
batch and use `stream()`/repeated grabs instead of per-frame `snapshot()`.
